## 1. Repository Pattern (Backend)

### 1.1 What It Does

The Repository pattern abstracts data access logic, hiding database details from services.

```
Without Repository:
SearchService → SQLite queries → SQL strings
Problem: Tight coupling, hard to test, hard to switch databases

With Repository:
SearchService → Repository interface → Multiple implementations
SearchRepository ─┬─ SQLiteRepository
                  ├─ PostgresRepository
                  └─ InMemoryRepository (for testing)
```

### 1.2 Your Implementation

Your backend correctly implements the Repository pattern:

**File**: [backend/src/repositories/base_repository.py](backend/src/repositories/base_repository.py#L25-L45)

```python
class BaseRepository(ABC, Generic[T]):
    """
    Abstract base repository for generic CRUD operations.
    
    Generic type T represents the entity type (Dataset, MetadataDocument, etc).
    Subclasses must implement abstract methods for database mapping.
    
    CRITICAL: Repositories do NOT auto-commit. Use UnitOfWork for transaction management.
    """

    def __init__(self, connection: sqlite3.Connection):
        self.connection = connection

    @property
    @abstractmethod
    def table_name(self) -> str:
        """Return the database table name."""
        pass

    @abstractmethod
    def _map_row_to_entity(self, row: sqlite3.Row) -> T:
        """Map database row to entity object."""
        pass

    @abstractmethod
    def _map_entity_to_dict(self, entity: T) -> Dict[str, Any]:
        """Map entity object to dictionary for insertion."""
        pass
```

## 2. Service Layer Pattern (Backend)

The Service layer orchestrates complex business logic, coordinating multiple repositories and infrastructure components.

```
Thin Service (Bad):
Request → Database → Response
(Just queries)

Fat Service (Our Pattern):
Request → Validate → Get dependencies → Execute → Transform → Respond
(Complex orchestration)
```

## 3. Factory Pattern (Document Extraction)

### 3.1 What It Does

Factory pattern creates objects of different types based on input, without client knowing which type is created.

```
Without Factory:
if file.endswith(".zip"):
    extractor = ZipExtractor()
elif file.endswith(".xml"):
    extractor = XMLExtractor()
else:
    extractor = DefaultExtractor()
Problem: Tight coupling, hard to add new types

With Factory:
extractor = ExtractorFactory.create(file)
Problem solved: Factory routes to right implementation
```

### 3.2 Your Implementation

Your backend uses factory pattern for document extraction:

- ✅ Routes to appropriate implementation
- ✅ Extensible for new types
- ✅ Hides implementation details

**Suggestion for A**: Use registry pattern for dynamic registration
```python
class ExtractorFactory:
    _registry: Dict[str, Type[DocumentExtractor]] = {}
    
    @classmethod
    def register(cls, extension: str, extractor_class: Type[DocumentExtractor]):
        cls._registry[extension] = extractor_class
    
    @classmethod
    def create(cls, file_path: str) -> DocumentExtractor:
        ext = Path(file_path).suffix.lower()
        extractor_class = cls._registry.get(ext)
        if not extractor_class:
            raise ValueError(f"No extractor for {ext}")
        return extractor_class()
```

## 4. Strategy Pattern (ETL Error Handling)

### 4.1 What It Does

Strategy pattern allows selecting different algorithms at runtime based on conditions.

```
Without Strategy:
if error_type == "network":
    retry with exponential backoff
elif error_type == "parsing":
    retry with linear backoff
elif error_type == "validation":
    fail immediately
Problem: Nested if-else hell

With Strategy:
strategy = RetryStrategy.EXPONENTIAL_BACKOFF
retry_handler.execute(operation, strategy)
Problem solved: Algorithm encapsulated in Strategy
```

### 4.2 Your Implementation

Your backend uses Strategy pattern for retry logic:

**File**: [backend/src/services/etl/etl_error_handler.py](backend/src/services/etl/etl_error_handler.py#L27-L45)

```python
class RetryStrategy(Enum):
    """Retry strategy options"""
    EXPONENTIAL_BACKOFF = "exponential_backoff"
    LINEAR_BACKOFF = "linear_backoff"
    FIXED_DELAY = "fixed_delay"
    NO_RETRY = "no_retry"


@dataclass
class RetryConfig:
    """Configuration for retry mechanism"""
    max_retries: int = 3
    initial_delay: float = 1.0  # seconds
    max_delay: float = 60.0  # seconds
    backoff_multiplier: float = 2.0
    strategy: RetryStrategy = RetryStrategy.EXPONENTIAL_BACKOFF
    jitter: bool = True  # Add randomness to prevent thundering herd
```

**Usage**:

```python
# Network errors: Aggressive retry
network_config = RetryConfig(
    max_retries=5,
    strategy=RetryStrategy.EXPONENTIAL_BACKOFF
)

# Parsing errors: Linear backoff
parsing_config = RetryConfig(
    max_retries=3,
    strategy=RetryStrategy.LINEAR_BACKOFF
)

# Validation errors: No retry
validation_config = RetryConfig(
    max_retries=0,
    strategy=RetryStrategy.NO_RETRY
)
```

### 4.3 Benefits

✅ **Flexibility**: Switch strategies based on error type
✅ **Testability**: Test each strategy independently
✅ **Maintainability**: Easy to add new strategies
✅ **Reusability**: Same retry logic used throughout ETL

### 4.4 Pattern Grade: A-

Your Strategy pattern is well-implemented:
- ✅ Enum for type safety
- ✅ Dataclass for configuration
- ✅ Multiple strategy options
- ✅ Easy to add new strategies

**Suggestion for A**: Implement as class hierarchy
```python
from abc import ABC, abstractmethod

class RetryStrategy(ABC):
    @abstractmethod
    def calculate_delay(self, attempt: int) -> float:
        pass

class ExponentialBackoff(RetryStrategy):
    def calculate_delay(self, attempt: int) -> float:
        return min(1.0 * (2 ** attempt), 60.0)

class LinearBackoff(RetryStrategy):
    def calculate_delay(self, attempt: int) -> float:
        return min(1.0 * attempt, 60.0)
```

```typescript
export class SearchQueryValidator {
    static readonly MIN_LENGTH = 1;
    static readonly MAX_LENGTH = 1000;

    static validate(query: string): void {
        const trimmed = query.trim();

        if (trimmed.length === 0) {
            throw new ValidationError('query', 'Search query cannot be empty');
        }

        if (trimmed.length > this.MAX_LENGTH) {
            throw new ValidationError('query', `Query cannot exceed ${this.MAX_LENGTH} characters`);
        }
    }

    static sanitize(query: string): string {
        return query.trim().replace(/\s+/g, ' ').slice(0, this.MAX_LENGTH);
    }
}

export class ChatMessageValidator {
    static readonly MIN_LENGTH = 1;
    static readonly MAX_LENGTH = 5000;

    static validate(message: string): void {
        const trimmed = message.trim();

        if (trimmed.length === 0) {
            throw new ValidationError('message', 'Message cannot be empty');
        }

        if (trimmed.length > this.MAX_LENGTH) {
            throw new ValidationError('message', `Message cannot exceed ${this.MAX_LENGTH} characters`);
        }
    }
}
```
