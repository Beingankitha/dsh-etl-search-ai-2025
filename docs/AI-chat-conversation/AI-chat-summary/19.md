## Prompt
> "Currently, stores directly call functions from `client.ts`. As I add Service and Repository layers, how should dependencies be injected? Should Iuse constructor injection, container-based DI, or functional closures? What pattern would best decouple our Svelte components from API implementation details?"

## Chat Answer

Dependency Injection (DI) is the foundation of loosely coupled, testable code. Your application demonstrates both excellent DI practices and room for improvement. This document validates your implementation and suggests enhancements.

## 1. The Problem: Tight Coupling Without DI

### 1.1 Tightly Coupled Code (Bad)

```typescript
// BAD: Component directly calls API
<script>
    import * as apiClient from '$lib/api/client';

    export let query = '';

    async function search() {
        const results = await apiClient.searchDatasets(query);  // Direct call
        // render results...
    }
</script>

<button on:click={search}>Search</button>
```

**Problems**:
- ❌ Component tightly coupled to apiClient
- ❌ Hard to test (can't mock apiClient)
- ❌ Can't swap implementations
- ❌ Hard to change API URL for different environments

### 1.2 Loosely Coupled Code (Good)

```typescript
// GOOD: Component receives service as dependency
<script>
    import type { SearchApiService } from '$lib/services/search.service';

    export let searchService: SearchApiService;
    export let query = '';

    async function search() {
        const results = await searchService.search(query);  // Use injected service
        // render results...
    }
</script>

<button on:click={search}>Search</button>
```

**Benefits**:
- ✅ Component doesn't know about implementation
- ✅ Easy to inject mock for tests
- ✅ Easy to swap implementations
- ✅ Easy to test in isolation

## 2. Your Current Implementation: Container-Based DI

### 2.1 The Container Pattern

Your frontend uses a centralized Container (Service Locator pattern):

**File**: [frontend/src/lib/container.ts](frontend/src/lib/container.ts)

```typescript
/**
 * Dependency Injection Container
 * 
 * Centralizes creation and configuration of all services and stores.
 * Provides:
 * - Single point of dependency management
 * - Easy mocking for tests
 * - Loose coupling between components
 * - Clear service registration
 */

export class Container {
    private httpClient: HttpClient;
    private searchService: SearchApiService;
    private chatService: ChatApiService;
    private searchStore: any;
    private chatStore: any;

    constructor() {
        // Initialize HTTP client
        this.httpClient = new HttpClient(API_BASE_URL, 30000, 3);

        // Initialize services
        this.searchService = new DefaultSearchApiService();
        this.chatService = new DefaultChatApiService();

        // Initialize stores
        this.searchStore = createSearchStore(this.searchService);
        this.chatStore = createChatStore(this.chatService);
    }

    // Getters for services and stores
    getHttpClient(): HttpClient {
        return this.httpClient;
    }

    getSearchService(): SearchApiService {
        return this.searchService;
    }

    getChatService(): ChatApiService {
        return this.chatService;
    }

    getSearchStore() {
        return this.searchStore;
    }

    getChatStore() {
        return this.chatStore;
    }

    /**
     * Update API base URL (useful for different environments)
     */
    setApiBaseUrl(url: string) {
        this.httpClient = new HttpClient(url, 30000, 3);
    }

    /**
     * Reset to initial state (useful for tests)
     */
    reset() {
        this.searchStore.clear();
        this.chatStore.clear();
        if (typeof (this.searchService as any).clearCache === 'function') {
            (this.searchService as any).clearCache();
        }
    }
}

// Create and export singleton container instance
export const container = new Container();
```

## 3. Dependency Injection Approaches

### 3.1 Constructor Injection (Purest)

Pass dependencies through constructor:

```typescript
// Service layer
export class DefaultSearchApiService implements SearchApiService {
    constructor(private httpClient: HttpClient) {}

    async search(query: string): Promise<SearchResponse> {
        return this.httpClient.request<SearchResponse>('/api/search', {
            body: { query }
        });
    }
}

// Store layer
export function createSearchStore(searchService: SearchApiService) {
    const store = writable<SearchState>(...);

    return {
        search: async (query: string) => {
            const results = await searchService.search(query);  // Uses injected service
        }
    };
}

// Component layer
<script>
    export let searchService: SearchApiService;  // Injected prop

    async function search() {
        await searchService.search(query);
    }
</script>
```

**Pros**:
- ✅ Explicit dependencies
- ✅ Easy to understand data flow
- ✅ TypeScript checks all dependencies

**Cons**:
- ❌ Lots of boilerplate for deep hierarchies
- ❌ Must pass through many layers

### 3.2 Container Pattern (Your Approach)

Centralized service locator:

```typescript
export const container = new Container();

// In component
<script>
    const searchService = container.getSearchService();
</script>
```

**Pros**:
- ✅ No prop drilling (don't pass through intermediate layers)
- ✅ Easy to reconfigure
- ✅ Good for large apps

**Cons**:
- ❌ Less explicit (where does dependency come from?)
- ❌ Harder to test without container modification

### 3.3 Functional Closures (Functional)

Create services with captured dependencies:

```typescript
// Factory function that creates service with dependencies
export function createSearchService(httpClient: HttpClient): SearchApiService {
    return {
        async search(query: string) {
            // httpClient captured in closure
            return httpClient.request('/api/search', { body: { query } });
        }
    };
}

// In container
export const searchService = createSearchService(httpClient);
```

**Pros**:
- ✅ Lightweight (no class overhead)
- ✅ Functional style matches SvelteKit
- ✅ Easy to compose

**Cons**:
- ❌ Less discoverable (where are methods defined?)

### 4.2 Backend Dependency Injection

Your backend uses dependency injection in services:

**File**: [backend/src/services/search/search_service.py](backend/src/services/search/search_service.py#L130-L155)

```python
class SearchService:
    def __init__(
        self,
        embedding_service: EmbeddingService,
        vector_store: VectorStore,
        unit_of_work: UnitOfWork,
        enable_caching: bool = True,
        cache_size: int = 1000
    ):
        """
        Initialize search service with dependencies.
        
        Args:
            embedding_service: EmbeddingService instance for query encoding
            vector_store: VectorStore instance for similarity search
            unit_of_work: UnitOfWork for repository access
            enable_caching: Enable LRU cache for query embeddings (optional)
            cache_size: Maximum number of cached query embeddings (default 1000)
        """
        # All dependencies injected, not created
        self.embedding_service = embedding_service
        self.vector_store = vector_store
        self.unit_of_work = unit_of_work
```

**Benefits**:
- ✅ Can inject mock embedding_service for testing
- ✅ Can inject different vector_store implementations
- ✅ Can test SearchService without database

### 4.3 FastAPI Dependency Injection

FastAPI uses built-in DI via `Depends()`:

**File**: [backend/src/api/routes/search.py](backend/src/api/routes/search.py)

```python
from fastapi import Depends, APIRouter

router = APIRouter(prefix="/api/search", tags=["search"])

# Define dependencies
async def get_search_service() -> SearchService:
    """Dependency: provides SearchService instance"""
    return SearchService(
        embedding_service=EmbeddingService(),
        vector_store=VectorStore(),
        unit_of_work=UnitOfWork()
    )

# Use dependencies
@router.post("/")
async def search(
    query: str,
    top_k: int = 10,
    search_service: SearchService = Depends(get_search_service)
) -> SearchResponse:
    """Search endpoint with dependency-injected service"""
    results = await search_service.search(query, top_k)
    return SearchResponse(results=results)
```

## 5. Improving Frontend DI: Constructor Injection Pattern

### 5.1 Current Approach (Container)


### 5.2 Better Approach (Constructor Injection)

```typescript
// In component
<script>
    import type { Readable } from 'svelte/store';
    import type { SearchState } from '$lib/stores/searchStore';

    export let searchStore: Readable<SearchState>;  // Injected from parent
</script>

{#each $searchStore.results as result (result.id)}
    <div>{result.title}</div>
{/each}
```

**In layout**:

```svelte
<script>
    import { container } from '$lib/container';

    const searchStore = container.getSearchStore();  // Get once
</script>

<SearchResults {searchStore} />  <!-- Pass as prop -->
```

**Benefits**:
- ✅ Component declares dependencies explicitly
- ✅ Easy to test with mock store
- ✅ Clear where data comes from
- ✅ Reusable in multiple places

### 5.3 Enhanced Container with Factory Methods

```typescript
// Better container pattern
export class Container {
    private httpClient: HttpClient;
    private searchService: SearchApiService;
    private chatService: ChatApiService;

    constructor(apiUrl: string = import.meta.env.VITE_API_URL) {
        // Lazy initialization
        this.httpClient = new HttpClient(apiUrl, 30000, 3);
        this.searchService = new DefaultSearchApiService(this.httpClient);
        this.chatService = new DefaultChatApiService(this.httpClient);
    }

    // Factory methods for better testability
    createSearchService(overrides?: Partial<SearchApiService>): SearchApiService {
        return overrides ? { ...this.searchService, ...overrides } : this.searchService;
    }

    createChatService(overrides?: Partial<ChatApiService>): ChatApiService {
        return overrides ? { ...this.chatService, ...overrides } : this.chatService;
    }

    // For testing: create mock container
    static createTestContainer(mocks: TestMocks): Container {
        const container = new Container('http://test-api.local');
        if (mocks.searchService) {
            container['searchService'] = mocks.searchService;
        }
        if (mocks.chatService) {
            container['chatService'] = mocks.chatService;
        }
        return container;
    }
}

// In tests
const mockSearchService = new MockSearchApiService();
const testContainer = Container.createTestContainer({ searchService: mockSearchService });
```

## 7. Decoupling Components from API Details

### 7.1 Current: Component Knows Too Much

```svelte
<!-- BAD: Component knows about API URLs, error types -->
<script>
    import * as apiClient from '$lib/api/client';
    import { ApiError } from '$lib/api/errors';

    async function search(query) {
        try {
            const response = await apiClient.searchDatasets(query, 10);  // Knows API
            results = response.results;
        } catch (error) {
            if (error instanceof ApiError && error.status === 401) {  // Knows error types
                // Handle auth error
            }
        }
    }
</script>
```

### 7.2 Better: Component Uses Service Abstraction

```svelte
<!-- GOOD: Component delegates to service -->
<script>
    import type { SearchApiService } from '$lib/services/search.service';

    export let searchService: SearchApiService;

    async function search(query) {
        try {
            const response = await searchService.search(query, 10);  // Uses service
            results = response.results;
        } catch (error) {
            // Service handles error translation
            errorMessage = error.message;
        }
    }
</script>
```

**Decoupling Benefits**:
- ✅ Component doesn't import apiClient
- ✅ Can swap search implementation (API → cache → mock)
- ✅ Error handling centralized in service
- ✅ Easy to test: inject mock service

| Approach | Coupling | Testability | Complexity | Best For |
|----------|----------|-------------|-----------|----------|
