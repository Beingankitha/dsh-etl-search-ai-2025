Your **base repository** uses class-based generic pattern:

```python
class BaseRepository(Generic[T], ABC):
    """Generic repository for data access abstraction."""
    
    def __init__(self, database: Database):
        self.database = database
    
    @abstractmethod
    def _map_row_to_entity(self, row: sqlite3.Row) -> T:
        """Map database row to domain entity."""
    
    @abstractmethod
    def _map_entity_to_dict(self, entity: T) -> Dict[str, Any]:
        """Map domain entity to database dict."""
    
    def get(self, id: int) -> Optional[T]:
        """Get single entity by ID."""
        row = self.database.query_one(
            f"SELECT * FROM {self.table_name} WHERE id = ?",
            (id,)
        )
        return self._map_row_to_entity(row) if row else None
    
    def find_all(self) -> List[T]:
        """Get all entities."""
        rows = self.database.query(f"SELECT * FROM {self.table_name}")
        return [self._map_row_to_entity(row) for row in rows]
    
    def insert(self, entity: T) -> Optional[int]:
        """Create new entity."""
        data = self._map_entity_to_dict(entity)
        return self.database.insert(self.table_name, data)
    
    def update(self, entity: T) -> bool:
        """Update existing entity."""
        data = self._map_entity_to_dict(entity)
        return self.database.update(self.table_name, data)
```

### 1.2 Unit of Work Pattern

**File**: [backend/src/repositories/unit_of_work.py](backend/src/repositories/unit_of_work.py)

```python
class UnitOfWork:
    """Unit of Work pattern - transaction management across repositories."""
    
    def __init__(self, database: Database):
        self.database = database
        self._repositories: Dict[str, BaseRepository] = {}
        
        # Lazy initialization of repositories
        self.dataset_repo: DatasetRepository
        self.metadata_doc_repo: MetadataDocumentRepository
        self.data_file_repo: DataFileRepository
        self.supporting_doc_repo: SupportingDocumentRepository
    
    def get_repo(self, repo_class: Type[T]) -> T:
        """Get or create repository instance."""
        repo_name = repo_class.__name__
        if repo_name not in self._repositories:
            self._repositories[repo_name] = repo_class(self.database)
        return self._repositories[repo_name]
    
    def commit(self) -> None:
        """Commit all changes."""
        self.database.commit()
    
    def rollback(self) -> None:
        """Rollback all changes."""
        self.database.rollback()
```

✅ **Transaction management**: Central point for coordinating multiple repositories

## 2. Class-Based vs. Function-Based

### 2.1 Comparison

| Aspect | Class-Based | Function-Based | Your Choice |
|--------|------------|-----------------|------------|
| **Encapsulation** | ✅ State + behavior | ❌ No state | ✅ Class |
| **Polymorphism** | ✅ Inheritance | ❌ Limited | ✅ Class |
| **Dependency Injection** | ✅ Constructor | ⚠️ Partial | ✅ Class |
| **Type Safety** | ✅ Generic types | ⚠️ Any | ✅ Class |
| **Testability** | ✅ Easy mocking | ✅ Easy mocking | ✅ Class |
| **Performance** | ⚠️ Instance overhead | ✅ No overhead | ✅ Class |
| **Code size** | ⚠️ More code | ✅ Less code | ✅ Class |

### 2.2 Your Implementation is Correct

**Class-based is right for**:
- ✅ Database abstraction layers
- ✅ Entities with shared behavior
- ✅ Transaction management
- ✅ Polymorphic behavior

**Function-based would be wrong for**:
- ❌ Can't share database connection
- ❌ Can't maintain state
- ❌ Can't use inheritance

## 3. Caching: Service vs. Repository

### 3.1 Current Architecture

Your **separation is correct**:

```
Frontend SearchApiService (handles caching)
    ↓
Backend API Route (no caching)
    ↓
SearchService (no caching, orchestrates search)
    ↓
Repository (database access, no caching)
```

### 3.2 Where Caching Goes

| Layer | Caching Type | Your Implementation |
|-------|--------------|-------------------|
| **Frontend** | HTTP response cache | ✅ SearchApiService.cache |
| **Service** | Business logic results | ❌ Not implemented (optional) |
| **Repository** | Database query cache | ❌ MetadataCache (separate) |
| **Database** | Query plan cache | ✅ SQLite handles |

**Recommendation**: Keep separation clear

```
SearchApiService       → Caches API responses (5-min TTL)
    ↓
SearchService          → No caching (stateless)
    ↓
MetadataCache          → Caches metadata documents (30-day TTL)
    ↓
Repository             → No caching (simple queries)
```

export function createSearchStore(searchService: SearchApiService) {
    const store = writable<SearchState>(initialState);

## 6. Repository Interface Design

### 6.1 Minimal Interface

Keep repositories focused:

```python
class BaseRepository(Generic[T], ABC):
    """Interface for data access."""
    
    # CRUD operations
    def get(self, id: int) -> Optional[T]: ...
    def find_all(self) -> List[T]: ...
    def insert(self, entity: T) -> Optional[int]: ...
    def update(self, entity: T) -> bool: ...
    def delete(self, id: int) -> bool: ...
    
    # Query helpers
    def count(self) -> int: ...
    def exists(self, id: int) -> bool: ...
```

### 6.2 Domain-Specific Methods

Add specialized queries in specific repositories:

```python
class DatasetRepository(BaseRepository[Dataset]):
    """Dataset-specific queries."""
    
    # Domain queries
    def get_by_identifier(self, file_identifier: str) -> Optional[Dataset]:
        """Find dataset by CEH identifier."""
        row = self.database.query_one(
            f"SELECT * FROM {self.table_name} WHERE file_identifier = ?",
            (file_identifier,)
        )
        return self._map_row_to_entity(row) if row else None

```

✅ **Good balance**: Generic interface + specific queries
