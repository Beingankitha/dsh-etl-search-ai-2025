## 1. Exponential Backoff Strategy

### 1.1 Why Exponential Backoff?

Exponential backoff prevents overwhelming a temporarily degraded server:
- **Linear backoff** (1s, 2s, 3s): Still hammers server during recovery
- **Exponential backoff** (1s, 2s, 4s, 8s): Gives server time to recover
- **With jitter** (±20%): Prevents thundering herd (all clients retrying at same time)

### 1.2 Frontend Implementation

Your frontend `HttpClient` implements exponential backoff correctly:

**File**: [frontend/src/lib/api/http-client.ts](frontend/src/lib/api/http-client.ts#L117-L130)

```typescript
private calculateBackoff(attempt: number): number {
    // Exponential backoff with max cap: 100ms, 200ms, 400ms, 800ms, max 5000ms
    // Prevents excessive delays on repeated failures
    const delayMs = Math.min(100 * Math.pow(2, attempt), 5000);
    // Add jitter (±20%) to prevent thundering herd
    const jitter = delayMs * 0.2 * (Math.random() - 0.5);
    return delayMs + jitter;
}

private sleep(ms: number): Promise<void> {
    return new Promise((resolve) => setTimeout(resolve, ms));
}
```

**Analysis**:
- ✅ Base delay: 100ms (suitable for frontend)
- ✅ Exponential multiplier: 2x each attempt
- ✅ Max cap: 5 seconds (prevents excessive delays)
- ✅ Jitter: ±20% randomization to desynchronize retries
- ✅ Calculation is deterministic and testable

## 2. Differentiated Retry Strategies by Error Type

### 2.1 Error Classification

Different errors require different handling:

| Error Type | Status | Retryable? | Retry Count | Reason |
|-----------|--------|-----------|-------------|--------|
| **Network Failures** | Network down | YES | 3-5 | Temporary, likely recoverable |
| **Timeout** | Network slow | YES | 2-3 | Usually transient |
| **5xx Server Errors** | Server degraded | YES | 2-3 | Server may recover |
| **429 Too Many Requests** | Rate limited | YES | 1-2 | Wait before retry |
| **4xx Validation Error** | Bad request | NO | 0 | Invalid data, won't change |
| **401 Unauthorized** | Auth failed | NO | 0 | Token expired/invalid |
| **403 Forbidden** | Access denied | NO | 0 | Permission issue |
| **404 Not Found** | Resource gone | NO | 0 | Endpoint doesn't exist |

### 2.2 Frontend Error Classification

Your frontend error module provides this classification:

**File**: [frontend/src/lib/api/errors.ts](frontend/src/lib/api/errors.ts)

```typescript
export function isRetryable(error: ApiError): boolean {
    // Don't retry validation or auth errors
    if (error.statusCode >= 400 && error.statusCode < 500) {
        // 429 (rate limit) is retryable
        if (error.statusCode === 429) return true;
        return false;  // Other 4xx not retryable
    }

    // Retry 5xx and network errors
    if (error.statusCode >= 500 || error instanceof NetworkError) {
        return true;
    }

    // Don't retry other errors by default
    return error.retryable === true;
}

export class ApiError extends Error {
    constructor(
        public statusCode: number,
        message: string,
        public retryable: boolean = true,
        public originalError?: Error
    ) {
        super(message);
    }
}

export class NetworkError extends ApiError {
    constructor(message: string, originalError?: Error) {
        super(0, message, true, originalError);
    }
}

export class TimeoutError extends ApiError {
    constructor(message: string) {
        super(0, message, true);
    }
}
```

**Usage in HttpClient**:

**File**: [frontend/src/lib/api/http-client.ts](frontend/src/lib/api/http-client.ts#L47-L62)

```typescript
async request<T>(
    endpoint: string,
    options: HttpRequestOptions = {}
): Promise<T> {
    const { method = 'GET', body, config = {} } = options;
    const timeout = config.timeout ?? this.defaultTimeout;
    const retries = config.retries ?? this.maxRetries;

    return this.requestWithRetry<T>(endpoint, method, body, timeout, retries, 0);
}

private async requestWithRetry<T>(
    endpoint: string,
    method: string,
    body: unknown,
    timeout: number,
    maxRetries: number,
    attempt: number
): Promise<T> {
    try {
        return await this.executeRequest<T>(endpoint, method, body, timeout);
    } catch (error) {
        const normalizedError = this.normalizeError(error);

        // Retry logic
        if (isRetryable(normalizedError) && attempt < maxRetries) {
            const delay = this.calculateBackoff(attempt);
            await this.sleep(delay);
            return this.requestWithRetry<T>(endpoint, method, body, timeout, maxRetries, attempt + 1);
        }

        throw normalizedError;
    }
}
```

```python
async def get(self, url: str, request_id: Optional[str] = None, **kwargs) -> dict:
    """GET request with retry logic and concurrency limiting."""
    await self._acquire_slot()
    try:
        headers = kwargs.pop("headers", {})
        if request_id:
            headers["X-Request-ID"] = request_id

        logger.info(f"GET {url}", extra={"request_id": request_id})
        async with self.session.get(url, headers=headers, **kwargs) as response:
            response.raise_for_status()  # Raises on 4xx/5xx
            data = await response.json()
            logger.info(f"GET {url} -> {response.status}", extra={"request_id": request_id})
            return data
    except asyncio.TimeoutError as e:
        logger.error(f"Timeout on GET {url}", extra={"request_id": request_id})
        raise HTTPTimeoutError(f"Request timeout: {url}") from e
    except aiohttp.ClientError as e:
        logger.error(f"HTTP error on GET {url}: {e}", extra={"request_id": request_id})
        raise HTTPClientError(f"HTTP error: {e}") from e
    finally:
        self._release_slot()
```

## 4. Errors That Should NOT Be Retried

### 4.1 Permanent Client Errors (4xx)

These indicate problems with the request that won't change:

| Code | Meaning | Why Not Retry |
|------|---------|---------------|
| **400** | Bad Request | Request format invalid, won't fix itself |
| **401** | Unauthorized | Auth token expired; need fresh login |
| **403** | Forbidden | Permission denied; same principal won't gain access |
| **404** | Not Found | Resource doesn't exist; won't appear |
| **409** | Conflict | State conflict; retry would likely fail identically |
| **422** | Unprocessable Entity | Validation failed; business rules violated |

## 6. Concurrency Control

### 6.1 Prevent Resource Exhaustion

Excessive concurrent requests can overwhelm frontend/backend:

**File**: [backend/src/infrastructure/http_client.py](backend/src/infrastructure/http_client.py#L84-L107)

```python
async def __aenter__(self):
    """Context manager entry."""
    self.session = aiohttp.ClientSession(timeout=self.timeout)
    self.semaphore = Semaphore(self.max_concurrent)  # Limit concurrent requests
    return self

async def _acquire_slot(self):
    """Acquire a slot from the semaphore to limit concurrent requests."""
    if not self.semaphore:
        raise HTTPClientError("Semaphore not initialized. Use 'async with' context manager.")
    await self.semaphore.acquire()

def _release_slot(self):
    """Release a slot back to the semaphore."""
    if self.semaphore:
        self.semaphore.release()
```

### 8.2 Suggested Improvements

**1. Circuit Breaker Pattern** (A-1)
```typescript

class CircuitBreaker {
    failureThreshold = 5;
    successThreshold = 2;
    timeout = 60000;  // milliseconds
    
    async execute<T>(fn: () => Promise<T>): Promise<T> {
        if (this.state === 'OPEN') {
            throw new Error('Circuit breaker is open');
        }
        // ...
    }
}
```

**Why**: Prevents waste of resources and user time during extended outages.

**2. Retry Metrics & Observability** (A-2)
```typescript
// Track retry patterns to identify systemic issues
interface RetryMetrics {
    endpoint: string;
    totalRequests: number;
    totalRetries: number;

    failedRetries: number;       // Retries that never succeeded
```

**Why**: Identifies problematic endpoints, helps tune retry parameters.

**3. Adaptive Retry Delays** (A-3)
```typescript
// Adjust backoff based on server response headers (Retry-After)
const retryAfter = response.headers.get('Retry-After');
if (retryAfter) {
    delay = Math.max(delay, parseInt(retryAfter) * 1000);
}
```

**Why**: Respects server's preferred retry timing, avoids wasted requests.

**4. Dead Letter Queue for Failed Retries** (A-4)
```typescript
// Store failed requests for manual retry or analytics
class FailedRequestQueue {
    async enqueue(request: Request, lastError: Error): Promise<void> {
        await db.failedRequests.insert({
            endpoint: request.url,
            body: request.body,
            error: lastError.message,
            timestamp: new Date(),
            attempts: request.retryCount
        });
    }
}
```

**Why**: Capture edge cases where all retries failed, enable recovery.
